# ビューアー モジュール分割（純粋連結方式）設計

日付: 2026-07-02 / 承認: 中村さん（案A採用）
対象: `Locahun3D_OfflineViewer.html`（22,680行 / 1.6MB / v0.2.0-beta系）

## 目的

**AI編集の信頼性・速度の改善。** 22K行の単一ファイルではClaudeがファイル全体を把握できず、grep往復・編集ミス・見落としが増える。機能ごとの小ファイル（800行以下）に物理分割し、編集を速く確実にする。

## 制約（不変条件）

- 配布形態は**単一HTMLファイル**のまま（オフライン納品・Worker配信・オンラインrepo同梱の3系統）
- 本リファクタで**コードの意味を1文字も変えない**。ESM化・リネーム・整理は一切しない（将来フェーズ）
- three.js 0.180 / Spark 2.0 / fflate のCDN importmapは現状維持
- 検証手段はブラウザ実機のみ（自動テストは存在しない）→ だからこそバイト一致検証を安全性の核にする

## 方式: 純粋連結（案A）

JSを既存の `// ══════` セクションマーカー（約45個）で断片ファイルに切り、ビルドスクリプトが**固定順で連結**して単一HTMLへ戻す。全断片は従来どおり1つの `<script type="module">` スコープを共有する。import/export化しないため、連結結果は元ファイルと完全に同一になる。

### リポジトリと置き場所

- 既存repo（github: Locahun3D/Locahun3D）を **`F:\Htlml\3DGS\Locahun3D\` に再クローン**し、以後の作業場所とする（新規repoは作らない）
- Dropbox内の現フォルダ（`F:\UNDEFINED Dropbox\...\3DGS\`）は「編集場所」から**ビルド成果物の受け皿**（他PC共有・Sanrio納品用）に降格。巻き戻り事故が起きても成果物コピーが壊れるだけでソースは無傷

### ソース構造

```
F:\Htlml\3DGS\Locahun3D\
  src/
    template.html      # HTML骨格。{{include:path}} プレースホルダ入り
    html/              # パネル・モーダル等の部分HTML（数ファイル）
    css/               # 5つの<style>ブロックをUI領域ごとに分割（~10ファイル）
    js/                # 約50ファイル。数字プレフィックスで連結順を明示
      010_state.js
      020_perf_tier.js
      030_renderer_scene.js
      ...
      9xx_main_loop.js
    assets/
      mixamo_glb.b64   # 埋め込みGLB(base64, ~400KB)を分離
  build.mjs            # 依存ゼロの連結スクリプト（Node単体）
  Locahun3D_OfflineViewer.html   # ビルド成果物。従来と同じパス
  viewer-dist/                    # 従来どおり（deploy-viewer.shが同期）
```

- 分割粒度: セクションマーカー単位。1,500行超の巨大セクション（カメラアニメ・FIGURE・カメラツール）は `// ──` サブセクション境界でさらに2〜3分割。**全ファイル800行以下**目標
- 順序管理は数字プレフィックスのみ。build.mjsはソート順連結（manifest不要）
- **CSSの5つの`<style>`ブロックは元の位置を維持する**（1つに統合しない）。template.htmlに5箇所のincludeポイントを置き、大きいブロックだけ複数ファイルへ分割してその位置で連結する。importmap・MIXAMO_GLB_B64のscriptタグも同様に元の位置のまま、中身だけincludeにする

### ビルド (build.mjs)

- `node build.mjs` — template.htmlのプレースホルダへ html/css/js/assets を差し込み、成果物を出力。npm依存ゼロ、数十ms
- エラー時は**即停止**: 断片欠落・空ファイル・未解決プレースホルダ
- `.build-hash` に前回ビルドの成果物ハッシュを記録。次回ビルド時に成果物が手編集されていたら**警告して停止**（`--force`で上書き）— 成果物への直接ホットフィックス消失を防ぐ
- 成果物先頭に「GENERATED — 編集禁止、ソースは src/」バナーを付与（移行完了後に追加）

### 移行手順（バイト一致が合格条件）

1. 切断位置リスト（出力ファイル名 → 元ファイルの行範囲）を作成。既存マーカーのgrepで機械的に決める
2. **使い捨て抽出スクリプト**がリストに従い元ファイルを行境界で切り出し、src/ を生成（人力コピペ禁止）。スクリプトは切断リストが**元ファイルの全行を隙間なく・重複なく被覆する**ことを検証してから実行する
3. `node build.mjs` → 出力と現行ファイルの **SHA-256一致を確認**（不一致なら切断位置を修正して再実行）
4. 一致確認後、GENERATEDバナーを追加（diffがバナーのみであることを確認）
5. ブラウザ実機スモーク: デモシーン読込・主要ツール開閉（カメラ/日照/測定/カメラワーク/ZIP保存）・スクショ検証
6. deploy-viewer.sh の先頭に `node build.mjs` を追加し、新リポジトリから deploy
7. メモリ（project_locahun3d_viewer）・運用手順の更新

### フロー更新（移行後の日常）

- 修正フロー: `src/` 編集 → `node build.mjs` → 実機確認 → commit+push → deploy
- オンラインrepo同期（public/viewer/offline-viewer.html）・Sanrio・Dropboxへのコピーは全て**成果物から**。コピー先パスは従来と同じ
- version.json の運用は無変更

## エラー処理

- ビルド失敗＝成果物を書き換えない（部分出力を残さない）
- 巻き戻り・破損検知: ソースはDropbox外＋git管理なので、成果物側が壊れてもビルドし直すだけ

## テスト方針

- 移行時: SHA-256一致（機械的証明）＋実機スモーク
- 以後: 従来の実機確認ルール（スクショ検証必須）を継続。単体テストはESM化フェーズ（将来）で導入
